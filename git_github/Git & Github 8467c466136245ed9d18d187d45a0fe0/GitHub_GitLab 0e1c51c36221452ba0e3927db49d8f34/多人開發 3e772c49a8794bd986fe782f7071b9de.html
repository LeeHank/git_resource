多人開發

  1. gitlab上開一個自己的專案(此時，自己的角色是maintainer，最高權限，可push到master分支與受保護分支)
  1. 到這個專案後，開個dev分支。之後，master分支就是只放版本號，dev分支就是開發過程用。並且在gitlab上，也把dev設為受保護的分支(去Settings/Repository/Protected Branches，把dev也設為push和merge都只有maintainer可以做)
  1. 在這個專案中，選setting/member，加Aaron進來，權限最高只給到developer(次高權限，無法push到master分支 & 受保護分支，只能push到未保護分支)
  1. 假設今天這個專案master已經走了一個commit(已發布初版了)，然後dev分支下面有2個commit，分別是Hank做的A功能與B功能。
  1. 現在，Aaron開始要為這個專案做事了。那流程會是這樣：
    1. Aaron先把這個專案clone到自己的主機
    1. clone下來後，會發現本機只幫你建了master分支，原本遠端的dev分支沒有看到。但果用git branch -a，會發現其實origin/dev還是存在的，只是沒幫你建到本地而已
    1. 此時，有兩種方式可以在本地建出dev：
      1. git branch dev origin/dev # 在本地自己設一個dev分支，並關聯到遠端的origin/dev分支
      1. git checkout -t origin/dev # 直接把遠端的origin/dev，複製到本機的dev分支，並且還幫你切到這個分支
    1. 有了這個分支後，他就會看到Hank已經把dev這個分支往前推進兩個commit了(分別是A功能和B功能)，此時，Aaron要來貢獻C功能
    1. 再建一個分支叫AaronFeatureC: git branch AaronFeatureC; git checkout AaronFeatureC
    1. 阿龍就在這個分支上做事，下commit，做完後，他想merge回dev分支，但先別這麼做…因為到時候你無法把dev分支推到gitlab上，dev是受保護分支。所以正確流程是，把AaronFeatureC直接推到gitlab上: git push -u origin AaronFeatureC #有寫-u，那之後他只要在本地的AaronFeatureC分支，寫git push，他就會自動幫你執行git push origin AaronFeatureC
    1. 那回到gitlab，就會看到現在多了一個AaronFeatureC的遠端分支了
    1. 接著，在gitlab上，選Merge Requests/Create merge request，並把source branch選AaronFeatureC，把Target branch選dev，Assignee選管理員，按submit，就送出merge request了
  1. 管理者在他的帳號下，就會看到Aaron發的Merge Request。
  1. 這時，管理者如果用眼睛看就可以做決定，那就可以點確認 or 拒絕並回覆他訊息
  1. 如果管理者用眼睛看不出來，而是需要上code跑跑看，那管理者就會在自己的本機，先checkout到dev分支，並下git fetch，得到origin/AaronFeatureC這個分支。然後用git log master ..origin/AaronFeatureC看一下commit訊息，然後也可以看內容
  1. 接著，就在dev分支下，寫 git merge origin/AaronFeatureC，那就會把這筆內容，merge到dev分支中。然後就可以在dev分支上做測試，看功能對不對
  1. 如果覺得hen不錯，那就可以把這個本地dev分支，push到遠端dev分支(這對管理者來說是可以的!): git push -u origin dev
  1. 那就完成了這一次的操作。
