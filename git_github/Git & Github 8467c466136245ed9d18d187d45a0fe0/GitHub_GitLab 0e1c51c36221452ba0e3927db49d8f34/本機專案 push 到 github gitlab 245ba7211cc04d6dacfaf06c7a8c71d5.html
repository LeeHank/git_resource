本機專案 push 到 github/gitlab

  • 這邊的情境是，在自己的本機上 (e.g. mac個人電腦 or Windows 個人電腦 or linux server 的個人帳號下)，已經有一個 git 專案，然後想把他同步到 github/gitlab 上
  • 那接下來要做的順序如下：

Case 1: 最單純的狀況 (本機端無分支、無tag)

Step 1: 在github/gitlab 上開一個遠端資料夾

  • 去github上，按New repository
  • Repository name可以取和本機git專案一樣的名稱，e.g. git-practice
  • 記得不要 initialize README file (因為一按，他會自動幫你加.git資料夾，加README file，然後幫你做第一次的commit。這時候就不是做一個空資料夾了，而是一個完整的.git資料夾，這會和本機端的.git資料夾打架。)
  • 按下create後，就得到這個專案的https路徑(or SSH路徑，e.g. git@github.com:hank/git-practice.git
  • (概念提醒：我們在github的帳號下開一個新資料夾，然後得到的https/ssh 路徑，就是在跟你說這個遠端資料夾的實際位置放在哪。那有了這個位址，本機就可以下指令說，要把本機的東西傳到這個新資料夾裡面了)

Step 2: 在本機端設定要連動的遠端資料夾

  • 告訴本機，這個遠端資料夾的位址，並設定nickname:
  • 在本機的 git 專案資料夾路徑下，去下這些 bash 指令：
# 先進到本機的專案資料夾路徑下
cd {本機的 git 專案資料夾路徑}

# 告訴本機，要連動的遠端資料夾位址，並幫他取個 nickname (通常都取 origin)
git remote add origin git@github.com:hank/git-practice.git

# 看一下現在 remote 的設定是不是如預期
git remote -v  

# 將本機的 git 專案傳上去
git push -u origin master # 把本機的 master，push 到 遠端的 origin 這個資料夾
  • 這邊提醒一下：
    • 第二行的 remote ，是只要跟遠端有關的指令，都會用 remote 來當起手式。
      • 像這邊就在 remote 後面加上 add ，表示要新增一個 remote 的位址
      • 下一行是在 remote 後面加上 -v ，表示要 view 一下目前 remote 的 settings
    • origin 在這邊是我們幫遠端資料夾路徑 git@github.com:hank/git-practice.git 所設的一個 nickname，之後只要打 origin ，他就知道指的是這個路徑。那更直覺的記法，是把 origin 直接想像成遠端資料夾的名稱。
    • 最後一行， -u 先不要看，先看最後一個字 master ，這是指 本機的 master 分支 。所以， push origin master 的意思是：我要把本機的 master 分支， push 到 origin 這個遠端資料夾(也就是剛剛設定好的位址)
    • (此時，遠端資料夾，會自動幫他新增一個分支，並沿用你的 master 這個名稱，將他取名為 origin/master 這個分支。這就就有點像shiny module的id前綴一樣，現在的前綴是origin/，這個"/"就是資料夾路徑的寫法了，origin的master)
    • 最後來講 -u， -u的全名是upstream，但不重要，重要的是 -u 就代表了一個代名詞，代替 origin master。所以之後可以直接寫 git push，然後後面啥都不寫，這時候，git就會猜測你是要把本機的 master 分支，推到 origin 上面。所以對於只有 master 分支的人，這個 -u 就顯得很方便了
    • 如果第一次沒有寫 git push -u origin master，而是寫 git push origin master，那之後每次push，也都得寫清楚，要把本機目前的分支，推到遠端資料夾的哪個分支，所以就得一直寫 git push origin master 了

Origin 概念細講

  • origin就是遠端資料夾的代名詞，他裡面帶有這個遠端資料夾位址(url)的資訊，方便我們做push和pull//fetch時，知道要去哪裡下載/上傳資料
  • 照上面的例子，當我把本地的 master 分支推到遠端時，其實就建立了一個遠端分支，叫做 origin/master (這是一個字，因為本機端已經有一個分支叫 master 了，所以遠端的這個分支，幫他加個前綴，叫他 origin/master 而已)。
  • 那既然 origin/master 是個遠端分支，那他就一樣只是一個貼紙，且這張貼紙現在指到最新的一筆commit。
  • 我們可用以下這段code來驗證這些概念：
# 先看一下遠端資料夾名稱，和對應的 url
git remote -v

# 本地分支放在 .git/refs/heads, 遠端分支放在 .git/refs/remotes
# 來看一下遠端分支裡面有啥東西：
ls .git/refs/remotes # 可以找到有 origin/master 這個檔案

# 看看 origin/master 這個檔案裡的內容
cat .git/refs/remotes/origin/master # 會看到裡面的內容就是一個SHA-1值
git cat-file -t {此SHA-1值} # 可以看到，他是個commit物件
git log --oneline # 可以看到，這個SHA-1，就是最新一筆的commit
                  # 且目前狀態寫成(HEAD -> master, origin/master)
                  # 表示，現在的HEAD指向本地的master分支，和遠端的origin/master分支
                  # 而這些分支現在全都指向這個最新的commit
